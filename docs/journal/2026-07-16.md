Fourth round: **anchor-based drop placement** replaced the index math in `ui/src/reorder.ts` (`anchorFor`/`rankedAfterDrop`/`positionAfterDrop`, 16 vitest cases). The drop resolves to "beside the card you dropped it against" — direction from display positions, computed against the *displayed* (possibly filtered) lane, then applied to the *full* lane order from the pre-drag server snapshot. This is what made dragging safe under filters (Michael's insight: dropping below a visible entry should just mean directly below it in the full order — hidden neighbors don't move); the drag-disabled-while-filtering guard is gone. Unranked or nil anchors remain tail drops (end-of-ranked-list). The anchor formulation reproduces the unfiltered behavior exactly, so it's one code path, not a filtered special case.

Fifth round: **item deletion — a recorded design change.** v1's spec said "the tool never deletes"; pilot use made Michael want the curation act as a gesture. The reasoning that makes it safe: removal was always *the operator's* act, and a tracked file's deletion is just a reviewable `D` in the diff — the judgment gate holds. The genuinely unprotected case is a never-committed item (no git history to restore from), so the UI's inline confirm names exactly that risk. Shape: `workspace.Delete` (hash-guarded, removes the file + all its order.yaml entries in one gesture — a deleted file's entries are prunable unconditionally), `POST /items/{filename}/delete`, trash icon in the item modal behind the confirm. **Agent guidance is unchanged** — the grimoire note's "never delete items" rule stands; this is an operator gesture. Rule text updated in vane's `AGENTS.md` with the design-change date. No CLI delete yet (batched curation from the terminal can come if wanted).
