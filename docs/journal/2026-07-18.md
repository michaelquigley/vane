# 2026-07-18 — search

Added title/body search. Michael asked whether it needed new infrastructure — it didn't, and the reason is worth keeping: the server already rebuilds from a fresh disk read on every request, so search is just a grep over the snapshot the board load already produces. No index, no store, nothing to go stale. `GET /search?q=` (case-insensitive substring over `Doc.Title` + the new `ItemDoc.Body()` accessor — everything after the fence, verbatim; fenceless documents are all body), returning matched filenames. The UI's header search box debounces 150ms and feeds the match set into `filterBoard` as one more composable predicate beside tags/subsystems/milestone; anchor-based dragging keeps working under it. Escape clears. If a pilot repo ever grows enough items that per-keystroke greps hurt, that's the moment search earns an index — not before.

**Design change: the lifecycle ends at evaluating.** Michael's pilot insight from the archive project: items are *prompts* — old-school development had stories, this practice has prompts, and vane is where prompts are designed and executed. A realized prompt doesn't park in a trophy lane; it gets deleted (the completion act the delete gesture now serves) with its information synthesized into the project first — the same close-out discipline as a realized spec leaving `docs/future/`. So `done` and `dropped` are retired: five states, `LaneOrder` and `ParseState` and the API enum all narrowed, board renders five lanes. Migration is honest and self-evident: existing `done`/`dropped` items parse as invalid-state → malformed-flagged in inbox, for re-triage or deletion. The grimoire note's states section carries the prompts framing and the synthesis-before-deletion discipline; the agent delete rule sharpened to "never on your own judgment; may perform when directed as part of close-out."
