## cross-lane drop reshuffle

Pilot friction from Michael: dragging a card from inbox to the bottom of horizon left the destination lane briefly mis-ordered at release, then everything animated into the intended positions — an awkward double movement. The mechanism: `handleDragOver` re-parents the card into the optimistic board at the index where it *crossed the lane border*, and same-lane hover after that is deliberately ignored — the moving gap under the pointer is pure dnd-kit transforms, with board state still holding the card at its entry slot. Release resets the transforms, so the entry-slot arrangement renders for a frame; the awaited `transition` response then repaints everything into place. The within-lane path never showed this because it repaints via `optimisticRanked` synchronously before awaiting — batched into the same render as the drag teardown, no intermediate frame.

The fix is the missing twin: `optimisticPlaced` (App.tsx, beside `optimisticRanked`) computes the drop from the pre-drag snapshot — card out of its origin lane, into the destination's ranked prefix at the anchored position — and `handleDragEnd`'s cross-lane branch now sets it synchronously before the await. The release frame shows the final arrangement; the server response matches it, so nothing moves twice. Unchanged by design: a drop into the unranked tail still serializes as end-of-ranked-list, so the card visibly snaps up past the unranked cards — but now once, at release, rather than as a post-response reshuffle.

## roadmap images

Pilot friction from another adopting repo: an item body referencing `images/…` inside the roadmap directory showed a broken image in the item modal. Two gaps, one on each side. The serve middleware had no route for roadmap files — anything that isn't `/api/*` or an embedded asset falls back to `index.html`, so the `<img>` received HTML. And react-markdown resolves relative urls against the page origin with no transform. The fix honors plain markdown semantics — a relative link resolves against the file's directory, exactly as Obsidian and GitHub read it: a read-only `/roadmap/*` route (`server.Assets`, mounted in serve.go beside the SPA middleware) serves the roadmap directory's files — fresh disk read per request, files only, directories and traversal 404 — and a `roadmapUrl` transform (`ui/src/markdown.ts`) prefixes relative urls with `/roadmap/` in the modal's renderer, composing with react-markdown's default dangerous-protocol sanitization. `.item-body img` got `max-width: 100%` so a large sketch can't blow out the modal. Tests on both sides: `TestAssets` (containment, directory 404, method guard) and five vitest cases for the transform (21 UI tests total now). The loader already skipped subdirectories, so `images/` was always legal to *store* — the convention note's Recognition section now says so explicitly (subdirectories hold assets, never items).

The terminus arc on this feature ran four rounds, all on the no-git boundary. Round one: the naive handler followed symlinks and would serve `.git` bytes through an escape — fixed with `os.Root` plus a `.git` component check. Round two sharpened both halves: an in-root symlink to a nested `.git` still passed, and the explicit `.git` string was itself a second product-code use beyond root discovery's one — fixed by rejecting every dot-prefixed component (`.git` never named) and refusing any symlink component via per-prefix `Lstat`. Round three: the roadmap directory itself could be a symlink at `.git` — fixed with a per-request `Lstat` on the root. Round four claimed the check-then-open sequence is racy — a concurrent process could swap a validated component for a symlink between syscalls — and asked for descriptor-relative no-follow opens. **Vetoed by Michael as noise floor**: this is a planning and documentation tool on localhost serving the operator's own files — there is no adversary to run a microsecond race — and the write path itself documents best-effort detection, no locks, git as the net; the read route doesn't owe stronger guarantees than the writes. Recorded here because terminus keeps no dispositions: if a future review re-raises the race, this is the veto to cite. The round-four proper-noun comment fix (Obsidian/GitHub) was applied. Final state: root verified non-symlink per request, `os.Root` confinement, no dot-prefixed component served, no symlink followed; `TestAssets` covers the escape, nested `.git`, in-root link, symlinked root, directory, and method cases.

Coda: "still not seeing my images" turned out to be the dev loop, not the feature — `ranger serve` served the image fine when hit directly, but the Vite dev server only proxied `/api`, so `/roadmap/*` fell into Vite's own index fallback. `/roadmap` added to the proxy in `vite.config.ts`; dev and prod now route identically. Worth remembering the diagnostic: the serve endpoint and the dev endpoint are different servers, and a route added to the middleware exists in dev only if the proxy forwards it.
