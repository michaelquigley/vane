# the UI and serve

`ranger serve --port N` (default 4114) presents the localhost board for one discovered root: bound to `127.0.0.1` only, repository-level fail-fast at startup (missing/unreadable roadmap directory, unreadable order.yaml), graceful shutdown on signal. It is the ad-hoc single-project instance of the same server the daemon runs — a synthesized one-project config through the same normalization, the same mux — see [daemon.md](daemon.md) for the resident, multi-project story and the error tiers.

The mux is project-scoped throughout. The ogen server mounts at `/api/v1` (every operation under `/projects/{project}` — see [api.md](api.md)); a small middleware routes `/api/*` there and serves the embedded SPA for everything else, falling back to `index.html` for client-side routes like `/p/{name}`. A third route, `/roadmap/{project}/…`, serves the named project's roadmap files read-only: the leading segment resolves through the project set, then the asset handler is unchanged from its single-root form — fresh disk read per request, files only, directory requests 404, root-confined (the root is verified non-symlink per request, and no symlink or dot-prefixed path component is ever served, so the route can never surface git metadata). Item bodies can thus reference images and attachments relative to the roadmap directory (`images/sketch.png`), the same reading Obsidian and GitHub give those links. `-tags no_ui` builds a headless binary whose middleware serves the API and says plainly that the board wasn't built in.

`ui/` follows the flo pattern: Vite + TypeScript + React 19, built into `dist/` and embedded via `go:embed all:dist` (builds not committed; `make build` runs the frontend first). The TypeScript client is generated from the same contract the server is — `npm run gen:api` runs `openapi-typescript` against `internal/api/specs/ranger.yml` into `src/api/schema.d.ts`, and all calls go through `openapi-fetch` in `src/api.ts` — so the server contract and the client types cannot diverge. `api.ts` is a project-bound factory: `makeApi(project)` returns the operation set with the project pre-applied (the object carries `project` for the modals and the markdown transform), and `fetchProjects()` is the one unscoped call. The Vite dev server proxies `/api` and `/roadmap` to a running serve or daemon for the hot-reload loop; `/p/{name}` needs no proxy (SPA-side). Everything is self-contained: the house fonts (Source Serif 4 and Source Code Pro, via bundled `@fontsource` packages), the Material Design icons (inlined SVG paths), and the binoculars-mark favicon all ride into the binary — no CDN fetch, fully offline.

## the project in the URL

The project lives in the URL — `/p/{name}` — so windows are independently addressable and bookmarkable, and multiple windows on different projects need no coordination. No router library: the SPA reads the path segment at load, and the bare `/` consults `GET /projects` and redirects to the default. Names are slug-shaped by the config grammar, so the segment is taken verbatim; no decoding rule exists anywhere.

The header carries a project selector in the project name's slot — a quiet native `<select>` fed by the project index. Available projects list by name, a dirty one suffixed with the git vernacular's own marker (`ranger *`, "uncommitted changes" in the tooltip) — and because the closed control renders the current option's label, the marker on the current project is the board-level glance for free, and the open list answers "is everything committed?" across every board at once. Unavailable projects stay present but disabled, the diagnostic in the label (`anpheq — roadmap directory not found`) and the tooltip — the same explain-don't-hide posture the board takes toward malformed items. A URL naming no configured project shows that name as a disabled entry, so the header reflects the URL truthfully. Switching is a navigation to `/p/{name}`, never a state mutation; everything downstream — board fetch, search, capture, asset URLs — keys off the URL's project. The index rides along with board loads and gesture outcomes — never the drag's optimistic repaints — so the markers and availability stay as fresh as the board.

The header and selector render from the project index, independent of any one board's success: a repository-level board failure is a body-region state under the live header — shown for unavailable and unknown projects alike — so a broken default never strands the operator away from healthy projects. Only the project *index* itself failing keeps the page-replacing fatal shape; that is the daemon-level tier, where there is genuinely nothing to select.

## the surface

Light theme, driven by CSS variables at the top of `src/style.css`.

- **Board** — five lanes in lifecycle order. Cards show the title (or the filename when the title is unreadable), the item's subsystems as outlined mono chips and tags as solid colored label chips (both sorted for display; file order untouched), its milestone as a small mono badge, and flag badges with the diagnostic on hover. A dashed rule marks each lane's ranked/unranked boundary. A card whose file is uncommitted in git carries a quiet amber left-edge accent (an inset shadow, so nothing shifts; the drag overlay keeps it) — information, not alarm — rendered only when the server sends a verdict, so a git-less root shows nothing rather than pretending cleanliness; see [api.md](api.md) for the verdict's semantics. Manual browser refresh is the freshness contract; no polling. Label colors live in `src/labels.ts`: the house vocabulary (defect, documentation, enhancement, epic, feature, spike, story, plus product labels) carries fixed colors matched to the practice's forge boards; any other tag derives a stable color from its text, so labels read consistently everywhere with zero configuration.
- **Filtering** — clicking a tag or subsystem chip narrows the board to cards carrying every selected one (forge-style AND); clicking a milestone badge narrows to that milestone (single-select), and the header's search box narrows to items whose title or body contains the query (server-side against a fresh read, debounced; Escape clears) — all of it composes. Active filters show in the header for removal. Lane counts and ranked boundaries follow the filtered view. Client-side presentation only; no API involvement. Drags stay live under filters because placement is anchor-based: the moved card lands directly beside the visible card it was dropped against in the lane's full order — hidden neighbors stay where they are — and a drop below an unranked card (or into empty space) is a tail drop, serializing as end-of-ranked-list. The gesture still computes against the pre-drag server truth.
- **Drag** — `@dnd-kit/core` + sortable, fully telegraphed: a floating copy of the card follows the pointer (`DragOverlay`), crossing into another lane live-moves the card so the destination opens a slot, and drops apply optimistically — the card stays where it lands while the gesture runs, and the server's fresh board replaces the preview when it returns. Gestures always compute against the pre-drag server-truth snapshot (hashes, order version, on-disk ranked counts), never the preview; a failed drop restores the snapshot and reloads. A within-lane drop PUTs the lane's resulting ranked prefix (an unranked card ranks alone; tail drops serialize as end-of-ranked-list); a cross-lane drop is transition-and-place.
- **Item modal** — click a card for a centered, fixed-size modal (`min(80vh, 56rem)` tall): the body renders as markdown (react-markdown + GFM), with a structured metadata block (labeled rows: state, created, milestone, subsystems, tags, source, file) and flags/log below the title. Relative urls resolve against `/roadmap/{project}/` so referenced images display; containment is a resolved-path check, never a spelling scan — the candidate resolves through real URL semantics and must sit under the current project's prefix, or it renders inert, because browsers normalize `../`, backslash separators, and percent-encoded dot segments alike before the request ever leaves. The default dangerous-protocol sanitization stays. Click the title to retitle in place (Enter commits the full two-file gesture, Escape/blur cancels). The trash icon deletes the item behind an inline confirm that names the no-git-history risk for uncommitted items. The pencil icon opens edit mode: the raw file bytes — frontmatter included — in a word-wrapped textarea, saved verbatim through the hash guard. `rename to slug` appears only on mismatch-flagged cards. Escape or a backdrop click closes.
- **Capture** — the plus icon in the header; title + optional body in a modal sized identically to the item modal, landing in the window's current project's inbox.
- **Conflicts** — every mutation resolves through one outcome path. `item_conflict`/`order_conflict`: a dismissable notice ("changed on disk — reloaded") and a board refetch. `slug_collision`: the capture modal and item modal keep their content on screen and show the structured recovery paths — the preserved `.capture-` draft for capture, source and colliding destination for the renames. Validation refusals show inline with nothing written. Partial two-file failure and any other fault surface the server's message verbatim.

## build and versioning

`make build` = the frontend (npm install + build) then `go install ./...`, per the archive repo's Makefile shape; `make headless` installs with `no_ui` and no frontend step; `make generate` regenerates both sides of the contract (ogen server code, TypeScript schema) from `specs/ranger.yml`; `make test` runs the Go suite uncached plus `go vet` (UI tests run via `npm test` in `ui/`); `make clean` removes the installed binary, `dist/`, and `node_modules/`; `make push` vendors the installed binary into the push depot.

Versioning follows the terminus pattern on `github.com/michaelquigley/push/build`: `ranger version` prints the figlet banner and build detail, ldflags-stamped for released builds, `v0.1.x [developer build]` otherwise.
